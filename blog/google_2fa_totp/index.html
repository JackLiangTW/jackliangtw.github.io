Google Authenticator / TOTP 2FA 運作原理與 Node 實作

運作原理(TOTP)

  • 伺服器與 App 共享一把 secret(用 QR Code 傳遞一次)。
  • 兩邊都用 HMAC-SHA1(secret, floor(time / 30s)) 算出 6 位 PIN。
  • 因為算法是純本地運算,驗證時不會打 Google API;Google 只是一個「內建這個算法的 App」。

實作步驟(後端)

  1. 使用者啟用 2FA → 後端產生 secret,存在該使用者欄位。
  2. 用 otpauth://totp/<issuer>:<account>?secret=<secret>&issuer=<issuer> 格式產 QR Code 給 App 掃。
  3. 使用者輸入 6 位碼 → 後端用同一 secret 重算對比。
  4. 允許 ±1 個時間窗誤差(前/後 30 秒各一)容忍手機時間飄移。

Node 範例:

import { authenticator } from 'otplib';

const secret = authenticator.generateSecret();          // 存 DB
const otpauth = authenticator.keyuri(userEmail, 'MyApp', secret);
// → 產 QR Code 給使用者

const isValid = authenticator.verify({ token: inputCode, secret });

踩到會痛的幾件事

  • Secret 是真正的 2FA 憑證——明碼存 DB 等於 2FA 失效。建議加密儲存或放 HSM / KMS。
  • QR Code 只顯示一次(或設定為只能顯示到 enable 前);使用者換手機一定要重新 enroll 或有備援 backup codes。
  • 伺服器時間飄走(NTP 沒跑好)會導致所有 2FA 失敗——排程任務要監控 ntpd / chronyd。
  • TOTP ≠ HOTP:HOTP 以 counter 遞增,TOTP 以時間為基準;都是 RFC 6238 / 4226,otplib 兩者都支援。
